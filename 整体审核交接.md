# 刀男大奥 · 整体审核交接(新 session 开场白 brief)

> 给「整体审核 session」的开场白。**canonical = 同目录 `设计文档.md`;本文件 = 审什么、怎么审、用什么 skill、别碰什么。**
> 用法:新 session 先跑 `/daonandaao` 对齐,再读本文件,然后开审。**审核 = 通读全文找问题、标记、分级 —— 不在审核中重设计**(修走 brainstorm)。

## 一句话
设计已大体完稿(§0–24 + 引擎层 + 3 个专题 brief),**切实现前做一次整体体检**:查 一致性 / 完整性 / 红线遵守 / 冗余 / 可实现性,产出**分级 findings**,再和作者决定修哪些。

## 为什么现在审
- 设计跨多 session、多次 append(§23 基建 / §24 章回 / §21 引擎层 a–e + ②⑤⑦ / Klaude prompt 转向),**section 间可能积累矛盾 / 漏洞 / 陈旧 §ref**。
- **切 plan / 写代码前**审一遍 = 最省返工(神隐挑战教训:边写边发现设计洞)。
- 已有审查先例(§21 用 writing-plans 范式通读、修 §9「算出」残留 + 定 schema 歧义)——这次范围扩到 **§0–24 全文 + 3 brief**。

## 用什么 skill(作者要求列出)
1. **`/daonandaao`** —— **第一步必跑**:载 HANDOFF + 设计文档 + 项目记忆,对齐锁定结论(审核基准)。
2. **`planning-with-files`（`/plan-zh` 或 `/pwf`）** —— **把审核做成 tracked 任务**:`task_plan.md`(审核维度 / 分段清单)+ `findings.md`(逐条问题 · 防漏防丢)+ `progress.md`。全文审一遍、findings 不会丢。**默认走这条**。
3. **（重型 · 最彻底 · 需 opt-in）Workflow 多 agent 编排** —— 大文档**并行审**(每 agent 认领一段或一维度 → 综合去重 → 对抗校验 → 分级)。**要显式开口**(说「ultracode」或「use a workflow」)才会跑;最快最全,适合一次性扫完 §0–24。
4. **`superpowers:brainstorming`** —— 审核**只标问题**;某洞要重设计时,审完逐个 fork brainstorm 修(**别在审核中边审边改**,防失焦)。

## 审什么(7 维度)
1. **内部一致性 / 矛盾**:跨 section 冲突(§17 机制 vs §23 基建 / §14.3 schema vs §21 zod / 同一「待决」两处结论不一)· 多次 append 的接缝。
2. **完整性 / 漏洞**:被引用却没设计的;**dangling 待决 / defer** 是否已挡实现;关键 path 全否(结局 / 逃跑残缺逃 / 溯行军坐实 / 触底死亡 / 重连三 gate)。
3. **锁定红线遵守**:逐 section 扫违反 —— 立场独立存储两轴 / 删读心 / 暗线无进度条 / **数值为正文服务** / 刀 100% 忠诚 ⊥ NPC 忠心 / 只 L1 进 schema / **prompt 以 Klaude 为范式(原件优先)** / 引擎拥真相 · AI 拥讲法 / 推手非死轨。(像上次 §9「算出」残留那种残骸。)
4. **冗余 / 可简化**:过繁 / 重复机制;违 §5 精简的。
5. **可实现性**:每机制 buildable 否(§20 T0 纯 TS 引擎核 / §21 确定性纯函数 · 可单测);有没有偷偷需要 LLM-in-loop 却归了引擎、或反之。
6. **§ref 健全**:cross-ref 指对地方否(全文海量 §ref · append / 未重编号后可能陈旧 · §23/§24 在 §22 之后但概念是机制细化⑤⑥)。
7. **正文优先 sanity**:整体真服务正文否 —— 有没有 mechanism-for-mechanism 的累赘(作者红线:数值为正文服务、正文才是主题)。

## 输出 = 分级 findings
按 **P0(矛盾 / 红线违反)· P1(漏洞 / 挡实现的 defer)· P2(冗余 / 可简化)· P3(§ref / 措辞)** 分级,每条 = 位置(§x.y / 行)+ 问题 + 建议。审完和作者过一遍,决定 **修(brainstorm)/ defer / 不动**。findings 落 `findings.md`(或审核报告),**不直接改 `设计文档.md`**。

## 锁定红线(审核纪律)
- **审核 ≠ 重设计**:只标问题,修走 brainstorm(一次一 fork、作者拍板)。
- **勿回退锁定结论**(HANDOFF「锁定结论」段)——发现的「问题」若其实是已拍板的取舍,标注「= 已锁、非 bug」别翻案。
- **读全**:`设计文档.md` §0–24 + 3 brief(`基建设计交接.md` · `UI设计交接.md` · `开局设计交接.md`)。
- **git 纪律**(踩过坑):findings / 审核报告单独成文、**别污染 `设计文档.md`**;真改 doc 时 `add` 指定文件别 `-A`(repo 有妖祀并行窗口)、`grep -c '^## 变更记录'` 应 = 1、插新章末尾接回 `---` + 变更记录。

## 方法 + 指针
`/daonandaao` → 读本文件 → `/plan-zh` 起审核任务(或 `ultracode` 走 Workflow 多 agent)→ 通读 §0–24 + 3 brief 填 `findings.md` → 分级 → 和作者过 → 修的走 brainstorm。**审完更新 `HANDOFF.md`**(审核结论 + 分级修复清单 + 下一步)。
