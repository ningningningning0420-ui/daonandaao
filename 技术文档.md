## T0. 总纲 · 核心 / 适配分层 🔒LOCKED

全卡技术架构的**第一原则**：

- **引擎核（Core）= 纯逻辑、平台无关**：持有 schema（当数据模型）+ 全部软护栏逻辑（夹幅度 / latent-cause 生命周期 / 派生复合量 / 失衡监测 / 软导演 / tell 指令生成）+ 管线编排 + 记忆 orchestration + 内容契约。**理想是纯 TS、可单测**（顺手接住设计复盘的 R7 可测试性）。
- **适配层（Adapter）= 薄、平台特定**：所有 **平台 I/O** 收口于此 —— 读写状态、发 LLM call、载内容、召回记忆。核心通过**接口**调适配层，不直接碰平台 API。

**三对三映射**（呼应 §4「账本 / 裁判 / 脾性」三分）：

| 设计层概念 | 核心负责 | 适配层负责 |
|---|---|---|
| 账本（状态） | schema 定义 + 读写逻辑 | 持久化（React: store / IndexedDB · ST: MVU） |
| 裁判（LLM） | 管线编排 + prompt 组装 | 发 call（React: 直接 API · ST: generateRaw） |
| 脾性（内容） | 内容契约（要哪些字段） | 载入（React: 数据文件 · ST: 世界书） |
| 记忆 / 上下文 | 分层 recall orchestration | 总结 / embed+检索 的实现 |

**为什么这么切**：
- **可移植**：核心是难的 90%、平台无关 → React 与 ST 复用同一引擎核，只换薄适配。这条让「React 先行 / 将来回酌 ST / Phase 2 搬服务端」三件事都只动适配、不动核心。
- **可测试**：核心是纯逻辑 → 可在浏览器 / 酒馆外单测（夹幅度、因果生命周期、派生量、记忆分层都是确定性函数）。
- **抗腐**：平台 I/O 不许渗进核心逻辑（= 别让引擎逻辑里散落 store / fetch / MVU 调用），否则核心被某平台锁死、移植性归零。

**平台 of record = React 先行 🔒**（详见 T3）。核心先行的回报在此兑现：**引擎核不动，只是主绑定走 React**。

## T1. 承载基质（岔口 1）🔒LOCKED = A · 混合（三分原则）

引擎逻辑 + 账本怎么分层 —— 三选项对应作者三张已有卡的栈：

- **A（采纳）· 混合**：**状态 / 逻辑 / 内容三分**，各落最擅长的层。
- B · 重型自建引擎（引擎自持一切状态）—— **落选**：重造持久 / swipe / 可检视，最重。
- C · 轻 / 声明式（自建代码压到最小）—— **落选**：程序化的软护栏（因果 arm→ripe→fire、软导演计数器）会跟声明式工具死磕 = 审神者「MVU 半残」之痛。

**采纳 A 的理由**：本卡软护栏是 procedural、stateful 的（C 扛不动），但又**比重型模拟轻**（账本 + 护栏），没必要把逻辑和状态焊死在一坨（B）。A = 每件事落在最擅长的层。

> **React 先行后（见 T3）**：A 的**混合原则**（状态 / 逻辑 / 内容三分）= 平台无关、就是 T0。「A」当初在 ST 语境下命名的专属绑定（MVU / 世界书 / build）**降级为将来可选**；**主绑定走下表 React 列**。

**适配层对照（同一核心，两套绑定）**：

| 核心要的 I/O | React 适配（主） | ST 适配（将来可选） |
|---|---|---|
| 状态持久化 | app store + IndexedDB | MVU 变量（chat metadata） |
| 发 LLM call | 直接 API（多 call · 缓存 · 流式） | generateRaw / 正常生成管线 |
| 载内容 | JSON / TS 数据文件 | 世界书 entries |
| 叙事总结 (b) | LLM call 自做 | 引擎 generateRaw 自做 |
| 向量召回 (c) | embeddings API + 向量 store（原生） | Vector Storage / 小白x 类插件 |

## T2. 记忆 / 上下文架构 🔒LOCKED

> 作者头号关切。核心立场一句话：**该持久的全是结构化 state、每回合确定性重注、不受 context 窗口限制；「上下文链接」靠结构化重注、不靠 transcript 存活 —— 引擎本身就是那根链。** ∴ 这游戏「长局会烂」的毛病天生比裸聊天 LLM 应用小一大截。

### T2.1 记忆有两类，先劈开

- **① 载重状态（账本）= 这游戏的「数据库」**：schema A–J 是引擎确定性读写的硬态，要求精确 / 可查 / 跨天不漂。体量极小（几十标量 + 两三数组，几 KB ~ 几十 KB）—— **本地 IndexedDB / app store 绰绰有余，不需要外部 DB / 向量库来「装得下」**。设计早把「必须被精确记住的」全 route 进 状态 + 史册（§14.3 + J）→ 载重记忆 **by construction 解决**。
- **② 叙事记忆（软态散文）= 会随 context 衰减的那块**：场景 texture、对话、emotional beats。设计没用账本管（§14.3），这才是「记忆系统」管的领域，且与 ① **正交**。

### T2.2 4 层模型

| 层 | 是什么 | 怎么解 |
|---|---|---|
| **L1 载重状态**（账本 A–J） | 国库 / 派系 / 每刀情契立场… | state，每回合确定性重注，与 context 窗口无关 |
| **L2 锁定史实**（史册 J） | 过去→现在的因果脊椎 | append-only，引擎挑相关注入（§14.4 史册↔因果） |
| **L3 叙事散文记忆** | 会衰减的 texture / 对话 / beats | 分层 recall（见 T2.4） |
| **L4 上下文组装 / 链接** | 每回合注什么、怎么排、不矛盾 | 引擎 = context librarian（岔口 5 prompt 组装） |

L1 / L2 设计已解、平台无关；L3 / L4 是记忆系统真正要建的。

### T2.3 史册 J 口径放宽 🔒

**史册 J 从「机械锁定事实」放宽到也收「情感 / 叙事里程碑」** —— 承诺、表白、背叛那种玩家会期待回扣的。**让最该被记住的回扣进结构化网、可靠重注**；只把环境 texture 留给衰减。每刀再挂一小撮「记忆锚」。

- 落账方式：裁判在 verdict 里除了机械事件，也可标记情感里程碑写入史册（沿用 §14.4 史册写入路径，扩 trigger 到情感转折）。
- 切口判断（已确认）：史册 **既装机械锁、也装情感锁**。
- 效果：大回扣（confession / 背叛）走 state、必中；小 texture（「锦鲤池边那句」）淡出、可接受。裁判靠 state + 脾性重建连贯情感连续性，不靠记得逐字散文。

### T2.4 分层叙事记忆 = 容三层、分阶实现 🔒

三层 **不是三选一，是叠起来的三层**，各管不同记忆、天生互补 = 长局连续性黄金标准：

- **(a) 结构化 state = 事实层**：always-on、精确、零 LLM 判断。（= L1 + 放宽后的 L2）
- **(b) 滚动总结 = 弧线层**：把淡出的散文压成「故事走到哪了」，让裁判 tonally / thematically 不跑偏。
- **(c) 向量召回 = 情景层**：把过去具体 beats embed，相关时捞回，做逐字回扣。

**架构「容三层」**：可插拔 **recall 接口返回分层 bundle**，引擎核统一管 **优先级 / 去重 / 配额 / 时效**（平台无关）；只有「怎么总结」「怎么 embed + 检索」落适配层。→ **三层全进核心契约，谁都没被现在就强迫实现。**

**实现「分阶上」**：先 (a)（放宽史册，立刻 ~80% 体感连续性）→ 加 (b) → 再 (c)，每加一层量体感真值。YAGNI 砍「一次全建」、不砍「为三层留好架构」。React 下 (b)(c) 皆原生（LLM call / embeddings API），无插件依赖。

### T2.5 orchestration 的真成本（「可行」≠「免费」）

建每层不难，**让它们别打架才是成本**：

- **冲突**：向量可能捞回与现态矛盾的旧散文 → chunk 打时间戳 + 被更晚史册推翻的丢弃 + **永远以 state 为 ground truth、召回只当「可能过时的过去」**。
- **冗余**：state + 史册 + 总结 + 向量 + 近 transcript 重复 → **分层优先级（state > 史册 > 总结 > 向量 > raw）+ 去重 + 每层配额**。
- **总结漂移**：长局总结套总结失真 → 分级总结（按弧）+ 锚在史册事实、不许和锁定真相矛盾。

### T2.6 记忆系统的安全红利 🔒

因硬 / 软切得干净，**秘密从不在散文里**（溯行军身份、因果在 schema E/H，叙事只露行为 tell、永不明说 §15.2）→ 任何总结 / 向量召回**顶多吸到行为 tell、吸不到真相** → 既不穿帮、反而帮续上 tell。**全要三层也不破这条。**
（注：这只保「叙事层」不泄密；**结构化 state 本身在纯前端可被 devtools 看** = 另一回事，见 T3 隐藏态完整性。）

## T3. 平台 of record & 部署拓扑 🔒LOCKED

**平台 of record = React 先行**（当**产品**做，非当卡）。核心 / 适配分层兑现：**React 适配为主绑定**（app store / 直接 LLM API / 数据文件 / 原生 embeddings）；**ST 绑定（T1 的 A）降级为「将来可选」**，引擎核不动。

**部署拓扑 = Phase 1 纯前端 SPA**（分阶 · 后端按需）：

- **LLM 接入 = BYOK**（玩家自带 API key）→ 无需后端代理 / 计费。
- **存档 / 账号 = 自带 · 纯本地**（IndexedDB；建议配 存档导出 / 导入 = 玩家自带存档）→ 无后端 DB、无账号系统。
- **隐藏态完整性 = Phase 1 接受可窥**：纯前端下 state 在浏览器、devtools 可见（溯行军 / 因果 / 立场会裸）。判断 = 顺 §15.6 非强迫精神「想自剧透是玩家自己的事」，Phase 1 不防窥。
  - 诚实边界：§15.6 本讲 in-fiction 读 tell，**meta 层 devtools 是延伸取舍** —— Phase 1 选择不处理。

**栈（Phase 1）**：Vite + React + IndexedDB，**引擎核跑在前端**，直连 LLM API（玩家 key），静态托管。最便宜最快、先验「好不好玩」。

**Phase 2 的缝（先留好、按需启用）**：作者拍板「先试纯前端、不行再调整」。当 **隐藏态被窥伤了体验 / 要面向无 key 用户托管 / 要云存档账号** 时 → 把**同一份引擎核搬服务端**（client 变薄渲染器 + 一道网络边界，核心不改）。核心 / 适配分层在此**第二次救场**。
